## 一、JavaScript 基础

### 1.1 原型链、__proto__ vs prototype

#### `prototype` vs `__proto__`

| | `prototype` | `__proto__` |
|---|---|---|
| 归属 | **函数**才有 | **所有对象**都有 |
| 作用 | 定义实例继承的属性/方法 | 指向创建该对象的构造函数的 `prototype` |

#### 关系图

```js
function Foo() {}
const obj = new Foo()

obj.__proto__ === Foo.prototype              // true
Foo.prototype.__proto__ === Object.prototype // true
Object.prototype.__proto__ === null          // 链的终点
```

#### 原型链查找规则

访问 `obj.xxx` 时，JS 引擎：
1. 先找 `obj` 自身属性
2. 没有 → 找 `obj.__proto__`（即 `Foo.prototype`）
3. 没有 → 找 `Foo.prototype.__proto__`（即 `Object.prototype`）
4. 没有 → `null`，返回 `undefined`

```js
function Animal(name) {
  this.name = name
}
Animal.prototype.speak = function () {
  return `${this.name} speaks`
}

const dog = new Animal('Rex')
dog.speak()                        // 'Rex speaks'（从原型链找到）
dog.hasOwnProperty('name')         // true（自身属性）
dog.hasOwnProperty('speak')        // false（在原型上）
dog.__proto__ === Animal.prototype // true
```

**关键记忆点：**
- `prototype`：函数的蓝图，`new` 时挂到实例的 `__proto__` 上
- `__proto__`：实例的指针，指向"父级"
- 推荐用 `Object.getPrototypeOf(obj)` 读取，不直接操作 `__proto__`

### 1.2 手写 new / instanceof / Object.create

#### 手写 `new`

```js
function myNew(Constructor, ...args) {
  // 1. 创建空对象，原型指向构造函数的 prototype
  const obj = Object.create(Constructor.prototype)
  // 2. 执行构造函数，绑定 this
  const result = Constructor.apply(obj, args)
  // 3. 若构造函数返回对象则用它，否则返回 obj
  return result instanceof Object ? result : obj
}

function Person(name) { this.name = name }
Person.prototype.greet = function() { return `Hi, ${this.name}` }

const p = myNew(Person, 'Tom')
p.greet()       // 'Hi, Tom'
p instanceof Person  // true
```

#### 手写 `instanceof`

```js
function myInstanceof(instance, Constructor) {
  if (typeof instance !== 'object' || instance === null) return false

  let proto = Object.getPrototypeOf(instance)
  while (proto !== null) {
    if (proto === Constructor.prototype) return true
    proto = Object.getPrototypeOf(proto)
  }
  return false
}

myInstanceof(p, Person)  // true
myInstanceof(p, Object)  // true（链上能找到）
myInstanceof(42, Number) // false（基本类型）
```

#### 手写 `Object.create`

```js
function myCreate(proto, propertiesObject) {
  if (typeof proto !== 'object' && typeof proto !== 'function') {
    throw new TypeError('proto must be object or null')
  }
  function F() {}
  F.prototype = proto
  const obj = new F()
  if (propertiesObject !== undefined) {
    Object.defineProperties(obj, propertiesObject)
  }
  return obj
}

const animal = { speak() { return 'animal sound' } }
const dog = myCreate(animal)
dog.speak()                           // 'animal sound'
Object.getPrototypeOf(dog) === animal // true
```

#### 三者关系

```
myNew 内部用 Object.create 设置原型
myInstanceof 沿着 __proto__ 链查找 Constructor.prototype
Object.create 是手动控制原型链的最底层工具
```

## 二、JS 执行机制

### 2.1 Event Loop（宏任务/微任务）& JS 单线程

#### 执行顺序口诀

```
同步代码 → 微任务队列清空 → 渲染 → 一个宏任务 → 微任务队列清空 → ...
```

#### 分类

| 类型 | 包含 |
|------|------|
| **微任务** | `Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver`、`async/await` |
| **宏任务** | `setTimeout`、`setInterval`、`setImmediate(Node)`、`I/O`、`MessageChannel` |

#### 经典例题

```js
console.log('1')
setTimeout(() => console.log('2'), 0)
Promise.resolve().then(() => console.log('3')).then(() => console.log('4'))
console.log('5')
// 输出：1 5 3 4 2
// 分析：同步1、5 → 微任务3 → 3完成产生4 → 微任务4 → 宏任务2
```

#### async/await 本质

```js
async function foo() {
  console.log('A')
  await Promise.resolve()   // await 后面 = .then() 回调，进微任务队列
  console.log('B')
}
console.log('start')
foo()
console.log('end')
// 输出：start A end B
```

#### JS 线程模型

```
浏览器进程
├── JS 主线程（单线程，跑你的代码）
├── Web Worker 线程（独立线程，无 DOM 访问）
├── Service Worker 线程
├── 渲染线程（与 JS 互斥，不能同时跑）
├── 网络线程
└── 定时器线程
```

```js
// Web Worker — 真正的多线程
const worker = new Worker('worker.js')
worker.postMessage({ data: bigArray })
worker.onmessage = (e) => console.log(e.data)

// worker.js（独立线程，不能访问 DOM/window）
self.onmessage = (e) => {
  const result = heavyComputation(e.data)
  self.postMessage(result)
}
```

| 问题 | 答案 |
|------|------|
| JS 主线程单线程？ | 是，永远只有一个调用栈 |
| 能并行执行 JS？ | 能，用 `Web Worker`（独立 V8 实例） |
| Worker 能操作 DOM？ | 不能，DOM 只属于主线程 |

### 2.2 宏任务 vs 微任务图示 & 为什么要设计两种类型

#### 完整 Event Loop 图示

```
┌─────────────────────────────────────────────────────┐
│                    Call Stack                        │
│         （同步代码在这里执行，LIFO）                  │
└─────────────────┬───────────────────────────────────┘
                  │ 执行完毕（栈空）
                  ▼
┌─────────────────────────────────────────────────────┐
│              Microtask Queue（优先）                  │
│   Promise.then │ queueMicrotask │ MutationObserver   │
│   ← 全部清空，包括执行过程中新加入的微任务 →           │
└─────────────────┬───────────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────────┐
│                  渲染（浏览器）                        │
│         requestAnimationFrame 在这里触发              │
└─────────────────┬───────────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────────┐
│              Macrotask Queue（取一个）                 │
│   setTimeout │ setInterval │ I/O │ MessageChannel    │
│   ← 每次只取队列头部 1 个执行 →                        │
└─────────────────┬───────────────────────────────────┘
                  └─ 执行完 → 回到顶部，再次清空微任务 → loop...
```

#### 为什么要设计两种类型？

| | 微任务 | 宏任务 |
|---|---|---|
| **目标** | 当前操作的"尾声"，保持状态一致性 | 独立的新任务，允许渲染插入 |
| **优先级** | 高，立即清空 | 低，一次一个 |
| **类比** | 饭后洗碗（必须马上做） | 明天的会议（排队等） |

- 只有宏任务：Promise 回调可能被定时器插队，UI 抖动
- 只有微任务：递归清空永不让出，页面卡死

#### Node.js Event Loop

```
timers → pending callbacks → poll → check(setImmediate)
⚠️ process.nextTick > Promise.then（Node 独有）
```

#### 其他平台对比

| 平台 | 类似机制 |
|------|---------|
| Python asyncio | `call_soon`（宏）vs `ensure_future`（微） |
| Dart/Flutter | `microtask queue` + `event queue`，几乎与 JS 一致 |
| Java | 多线程，无"微任务"概念 |
| Rust (Tokio) | Waker + Poll 模型 |

## 三、JS 异步

### 3.1 async/await 原理（Generator + 自动执行器）

#### Generator 基础

```js
function* gen() {
  const a = yield Promise.resolve(1)  // 暂停，把 Promise 交出去
  const b = yield Promise.resolve(2)  // 再次暂停
  return a + b
}
// 手动调用：g.next() → g.next(value) → g.next(value)
// 问题：手动 .next() 太麻烦 → 需要自动执行器
```

#### 手写自动执行器（co 核心）

```js
function co(genFn) {
  return new Promise((resolve, reject) => {
    const g = genFn()

    function step(nextFn) {
      let result
      try {
        result = nextFn()
      } catch (e) {
        return reject(e)
      }

      if (result.done) return resolve(result.value)

      Promise.resolve(result.value)
        .then(val => step(() => g.next(val)))    // 成功：传值继续
        .catch(err => step(() => g.throw(err)))  // 失败：向内部抛错
    }

    step(() => g.next())
  })
}
```

#### async/await 等价关系

```js
// 你写的
async function fetchData() {
  const user = await getUser()
  const posts = await getPosts(user.id)
  return posts
}

// 编译器等价于（概念）
function fetchData() {
  return co(function* () {
    const user = yield getUser()
    const posts = yield getPosts(user.id)
    return posts
  })
}
```

#### 原理图示

```
async function
  ↓ await Promise
  暂停（= yield）→ 进微任务队列
  Promise resolve → .next(value) 恢复
  ↓ return result
  外层 Promise resolve
```

> V8 实际不是字面转 Generator，而是直接生成状态机字节码，概念模型一致，性能更好。

## 四、作用域 & 闭包

### 4.1 闭包 & 作用域

#### 作用域基础

```js
var a = 1       // 函数作用域
let b = 2       // 块级作用域
const c = 3     // 块级作用域

function foo() {
  if (true) {
    let y = 'only in block'   // 块级
    var z = 'hoisted to foo'  // 函数级
  }
  console.log(z)  // ✅
  console.log(y)  // ❌ ReferenceError
}
```

**词法作用域**：作用域在**写代码时**就确定，和调用位置无关。

#### 闭包定义

> **函数 + 它能访问的外部变量 = 闭包**
> 本质：函数记住了自己被**定义时**的作用域环境

```js
function makeCounter() {
  let count = 0
  return {
    inc: () => ++count,
    dec: () => --count,
    get: () => count
  }
}
const counter = makeCounter()
counter.inc()  // 1
counter.inc()  // 2
// count 不会被 GC，因为闭包持有它的引用
```

#### 经典陷阱：循环 + 闭包

```js
// ❌ 全部输出 5（var 无块级作用域，所有回调共享同一个 i）
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0)
}

// ✅ let 天然安全（每次迭代有独立的块级 i）
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0)  // 0 1 2 3 4
}
```

#### 实际应用

```js
// 防抖（经典闭包）
function debounce(fn, delay) {
  let timer = null          // timer 被闭包持有
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}

// 柯里化
const add = (a) => (b) => a + b
const add5 = add(5)
add5(3)  // 8

// 数据私有化
function createStore(init) {
  let state = init
  return {
    get: () => state,
    set: (val) => { state = val }
  }
}
```

#### 内存泄漏注意

```js
function attachEvent() {
  const bigData = new Array(1000000).fill('x')
  document.addEventListener('click', () => {
    console.log(bigData.length)  // 闭包持有 bigData，不被 GC
  })
  // 不移除监听器 → 泄漏
}
```

### 4.2 闭包内存持有：ECMAScript 规范视角

#### 规范核心概念

```
执行上下文 (Execution Context)
└── 词法环境 (Lexical Environment)
    ├── 环境记录 (Environment Record)  ← 真正存变量的地方
    └── outer（指向外层词法环境）

函数对象
└── [[Environment]]  ← 指向函数被"定义时"所在的词法环境
```

#### 引用链图示

```js
function makeCounter() {
  let count = 0
  return { inc: () => ++count, get: () => count }
}
const counter = makeCounter()
```

```
counter（GC Root，用户代码可达）
  ├─▶ counter.inc（函数对象）
  │       └── [[Environment]] ──────────┐
  └─▶ counter.get（函数对象）            │
          └── [[Environment]] ──────────┤
                                        ▼
                           makeCounter 的词法环境
                           ┌──────────────────────┐
                           │  Environment Record   │
                           │  ┌─────────────────┐ │
                           │  │ count = 2  ←这里 │ │
                           │  └─────────────────┘ │
                           └──────────────────────┘
```

#### 规范原文

> ECMA-262 §10.2.3 OrdinaryFunctionCreate:
> *"Set F.[[Environment]] to the running execution context's LexicalEnvironment."*

只要 `inc/get` 函数对象可达 → `[[Environment]]` 引用的 Environment Record 不被 GC → `count` 不被回收。

#### DevTools 验证

Memory → Heap Snapshot → 搜索 `closure` → 找到 `context: makeCounter` → 里面有 `count`。DevTools 把 `[[Environment]]` 展示为 `context`。

### 4.3 暂时性死区（TDZ）

#### 是什么

`let`/`const` 声明从**块作用域开始到声明语句执行之前**，访问会抛 `ReferenceError`，这段区域叫 TDZ。

```js
{
  console.log(x)  // ❌ ReferenceError（TDZ）
  let x = 1       // TDZ 结束
  console.log(x)  // ✅ 1
}
```

#### 与 `var` 对比

| 阶段 | `var` | `let` / `const` |
|------|-------|-----------------|
| 提升 | ✅ 提升 | ✅ 也提升（但不初始化） |
| 初始化 | `undefined` | ❌ 未初始化（TDZ） |
| 可访问 | 提升后立即 | 声明语句执行后 |

#### let 也提升的证明

```js
let x = 'outer'
{
  console.log(x)  // ❌ ReferenceError，不是 'outer'
  let x = 'inner' // 如果不提升，上面应访问到 outer
}
// 内层 x 确实提升了，只是处于 TDZ
```

#### 规范三状态

```
CreateBinding（提升，TDZ）→ InitializeBinding（声明执行）→ SetMutableBinding（赋值）
```

#### 常见陷阱

```js
// 函数默认参数（从左到右求值）
function foo(a = b, b = 1) {}
foo()  // ❌ ReferenceError：b 在 TDZ 中

// class 也有 TDZ
const obj = new MyClass()  // ❌
class MyClass {}
```

#### 为什么设计 TDZ？

让"先用后声明"的 bug **快速失败、显式报错**，代码更可预测。

## 五、高频手写题

### 5.1 高频手写题全集

#### 防抖 Debounce

```js
function debounce(fn, delay, immediate = false) {
  let timer = null
  return function (...args) {
    if (timer) clearTimeout(timer)
    if (immediate && !timer) {
      fn.apply(this, args)  // 首次立即执行
    }
    timer = setTimeout(() => {
      if (!immediate) fn.apply(this, args)
      timer = null
    }, delay)
  }
}
```

#### 节流 Throttle

```js
// 时间戳版（首次立即触发）
function throttle(fn, interval) {
  let last = 0
  return function (...args) {
    const now = Date.now()
    if (now - last >= interval) {
      last = now
      fn.apply(this, args)
    }
  }
}

// 定时器版（最后一次会触发）
function throttle(fn, interval) {
  let timer = null
  return function (...args) {
    if (!timer) {
      timer = setTimeout(() => {
        fn.apply(this, args)
        timer = null
      }, interval)
    }
  }
}
```

#### 深拷贝 Deep Clone

```js
function deepClone(target, map = new WeakMap()) {
  if (target === null || typeof target !== 'object') return target
  if (map.has(target)) return map.get(target)  // 处理循环引用
  if (target instanceof Date) return new Date(target)
  if (target instanceof RegExp) return new RegExp(target)

  const clone = new target.constructor()
  map.set(target, clone)

  if (target instanceof Map) {
    target.forEach((val, key) => clone.set(deepClone(key, map), deepClone(val, map)))
    return clone
  }
  if (target instanceof Set) {
    target.forEach(val => clone.add(deepClone(val, map)))
    return clone
  }

  ;[...Object.keys(target), ...Object.getOwnPropertySymbols(target)]
    .forEach(key => { clone[key] = deepClone(target[key], map) })
  return clone
}
```

#### 柯里化 Curry

```js
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args)           // 参数够了，执行
    }
    return function (...args2) {
      return curried.apply(this, args.concat(args2))  // 继续收集
    }
  }
}

const add = (a, b, c) => a + b + c
const curriedAdd = curry(add)
curriedAdd(1)(2)(3)   // 6
curriedAdd(1, 2)(3)   // 6
```

#### Array.flat

```js
Array.prototype.myFlat = function(depth = 1) {
  if (depth === 0) return this.slice()
  return this.reduce((acc, val) => {
    if (Array.isArray(val)) acc.push(...val.myFlat(depth - 1))
    else acc.push(val)
    return acc
  }, [])
}
```

#### call / apply / bind

```js
Function.prototype.myCall = function(ctx, ...args) {
  ctx = ctx == null ? globalThis : Object(ctx)
  const fn = Symbol('fn')
  ctx[fn] = this
  const result = ctx[fn](...args)
  delete ctx[fn]
  return result
}

Function.prototype.myApply = function(ctx, args = []) {
  ctx = ctx == null ? globalThis : Object(ctx)
  const fn = Symbol('fn')
  ctx[fn] = this
  const result = ctx[fn](...args)
  delete ctx[fn]
  return result
}

Function.prototype.myBind = function(ctx, ...outerArgs) {
  const fn = this
  function bound(...innerArgs) {
    return fn.apply(this instanceof bound ? this : ctx, [...outerArgs, ...innerArgs])
  }
  bound.prototype = Object.create(fn.prototype)
  return bound
}
```

#### 发布订阅 EventEmitter

```js
class EventEmitter {
  constructor() { this._events = {} }

  on(event, listener) {
    ;(this._events[event] ??= []).push(listener)
    return this
  }

  once(event, listener) {
    const wrapper = (...args) => { listener(...args); this.off(event, wrapper) }
    wrapper._origin = listener
    return this.on(event, wrapper)
  }

  off(event, listener) {
    if (!this._events[event]) return this
    this._events[event] = this._events[event]
      .filter(l => l !== listener && l._origin !== listener)
    return this
  }

  emit(event, ...args) {
    ;(this._events[event] ?? []).forEach(l => l(...args))
    return this
  }
}
```

#### LRU 缓存

```js
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity
    this.cache = new Map()  // Map 的插入顺序 = 访问顺序
  }

  get(key) {
    if (!this.cache.has(key)) return -1
    const val = this.cache.get(key)
    this.cache.delete(key)
    this.cache.set(key, val)  // 移到末尾（最近使用）
    return val
  }

  put(key, value) {
    if (this.cache.has(key)) {
      this.cache.delete(key)
    } else if (this.cache.size >= this.capacity) {
      this.cache.delete(this.cache.keys().next().value)  // 删头部（最老）
    }
    this.cache.set(key, value)
  }
}
```

#### 关键考点速查

| 题目 | 核心点 |
|------|-------|
| 防抖 | `clearTimeout` + `apply` 保持 this |
| 节流 | 时间戳 vs 定时器，首尾触发差异 |
| 深拷贝 | `WeakMap` 处理循环引用，保持原型 |
| 柯里化 | `fn.length` 判断参数是否够 |
| flat | `reduce` 递归 or 栈迭代 |
| bind | `instanceof` 检测 `new` 调用 |
| 发布订阅 | `once` 用包装函数 + `_origin` 标记 |
| LRU | `Map` 利用插入顺序模拟双向链表 |

### 5.2 LRU 缓存原理

**LRU = Least Recently Used（最近最少使用）**：满了淘汰**最久没被用过的**。

```
容量 = 3：
put(1) → [1]
put(2) → [1, 2]
put(3) → [1, 2, 3]  满了
get(1) → 1变最新  [2, 3, 1]
put(4) → 淘汰2  [3, 1, 4]
get(2) → -1（已淘汰）
```

Map 天然有序：头部 = 最老，尾部 = 最新，O(1) 读写。

**使用场景：** 浏览器 HTTP 缓存、React 组件缓存、DNS 缓存、Redis `allkeys-lru`、CPU L1/L2 Cache。

## 七、React 18 新特性

### 7.1 useTransition / useDeferredValue

#### 低优先级是什么意思

```
紧急更新（Urgent）：用户输入、点击 → 必须立即响应
过渡更新（Transition）：搜索结果、列表筛选 → 可以被打断/延迟

低优先级 = 可以被打断
用户还在输入时，未完成的过渡渲染会被丢弃，重新开始
```

#### useTransition — 你控制"哪个 setState 是低优先级"

```js
const [isPending, startTransition] = useTransition()

function handleInput(e) {
  setInput(e.target.value)              // 紧急：立即更新
  startTransition(() => {
    setList(search(e.target.value))    // 低优先级：可被打断
  })
}

{isPending && <Spinner />}
```

#### useDeferredValue — 你控制"哪个值延迟跟随"

```js
function SearchResults({ query }) {
  const deferredQuery = useDeferredValue(query)  // 滞后版本

  const results = useMemo(() => search(deferredQuery), [deferredQuery])
  const isStale = query !== deferredQuery  // 是否还在追赶

  return (
    <div style={{ opacity: isStale ? 0.5 : 1 }}>
      <List data={results} />
    </div>
  )
}
```

#### 核心区别

| | `useTransition` | `useDeferredValue` |
|---|---|---|
| **控制点** | setState 调用处 | 值的消费处 |
| **适用** | 你能改的 state | 外部传入的 prop/value |
| **pending 状态** | ✅ `isPending` | ❌ 自己比较 `val !== deferredVal` |

**选择原则：**
- state 在你手里 → `useTransition`（wrap 那个 setState）
- value 从外面传进来 → `useDeferredValue`（wrap 那个 prop）

#### 底层机制

两者都把更新标记为 `TransitionLane`（低优先级泳道），React 优先处理 `SyncLane`（紧急），后处理 `TransitionLane`。

### 8.2 useFormStatus 如何感知父 form

#### 核心机制：隐式 Context

`useFormStatus` 本质是 React 内部维护了一个**隐式 Context**，`<form action={...}>` 充当 Provider。

```js
// 概念实现
const FormStatusContext = createContext({ pending: false })

function ReactForm({ action, children, ...props }) {
  const [status, setStatus] = useState({ pending: false })

  async function handleSubmit(e) {
    e.preventDefault()
    const formData = new FormData(e.target)
    setStatus({ pending: true })
    try {
      await action(formData)
    } finally {
      setStatus({ pending: false })
    }
  }

  return (
    <FormStatusContext.Provider value={status}>
      <form onSubmit={handleSubmit} {...props}>{children}</form>
    </FormStatusContext.Provider>
  )
}

function useFormStatus() {
  return useContext(FormStatusContext)
}
```

#### 就近原则（同 Context）

```
<form action={a1}>             ← Provider（a1 状态）
  <form action={a2}>           ← Provider（a2 状态）
    <SubmitButton />           ← 读 a2 的状态
  </form>
  <SubmitButton />             ← 读 a1 的状态
</form>
```

#### 关键约束

```jsx
// ❌ form 外使用：pending 永远 false
// ❌ form 组件自身内使用：Provider 只对子树生效
// ✅ form 内部的子组件里使用
```

### 8.3 use(promise) 未 resolve 前组件如何渲染

#### 核心机制：throw Promise

```
use(userPromise) 在 Promise pending 时 → throw 这个 Promise
React 捕获 throw 的 Promise → 停止渲染 → 显示最近的 Suspense fallback
Promise resolve → 重新渲染子组件 → use() 返回缓存值 → 正常渲染
```

#### use() 内部伪代码

```js
function use(promise) {
  if (promise.status === 'fulfilled') return promise.value
  if (promise.status === 'rejected') throw promise.reason

  if (promise.status === 'pending') throw promise  // 挂起！

  // 首次调用：打标记，监听结果
  promise.status = 'pending'
  promise.then(
    value => { promise.status = 'fulfilled'; promise.value = value },
    reason => { promise.status = 'rejected'; promise.reason = reason }
  )
  throw promise
}
```

#### 执行流程

```
React 渲染 UserCard
  ↓ 执行到 use(userPromise)，Promise pending
  ↓ throw userPromise
  ↓ return 那行【永远不会执行】
  ↓ React 向上找最近的 Suspense
  ↓ 渲染 fallback={<Spinner />}
  ↓ 订阅 promise.then()
  ↓ Promise resolve
  ↓ 触发重新渲染
  ↓ use(promise) 返回缓存结果（不再 throw）
  ↓ return <div>{user.name}</div> ✅
```

| 阶段 | UserCard 执行 |
|------|--------------|
| Promise pending | 执行到 `use()` 中断，`return` 不执行 |
| Promise resolve | 整个组件**重新执行**，`use()` 返回值 |
| Promise reject | throw error，ErrorBoundary 捕获 |

> 这种模式叫 **"Algebraic Effects 的模拟实现"**，组件被中断后可从头重新执行，像从未被中断过。

## 九、React Server Components（RSC）

### 9.1 RSC 全面讲解

#### 背景：传统 CSR 的痛点

```
白屏时间长 → JS Bundle 巨大 → 数据瀑布 → 敏感逻辑暴露在客户端
SSR 解决了首屏，但 hydration 仍需下载全部 JS
```

#### RSC 核心思想

> **让组件在服务端运行，只把结果发给客户端，而不是把代码发给客户端**

#### Server Component vs Client Component

| | Server Component | Client Component |
|---|---|---|
| **声明** | 默认（无标记） | 文件顶部 `'use client'` |
| **运行环境** | 服务端（Node.js） | 客户端（浏览器） |
| **打包进 Bundle** | ❌ | ✅ |
| **直接访问 DB/FS** | ✅ | ❌ |
| **useState/useEffect** | ❌ | ✅ |
| **事件处理** | ❌ | ✅ |

```jsx
// Server Component（直接查 DB，不打包进客户端）
async function ProductPage({ id }) {
  const product = await db.products.findById(id)
  return (
    <div>
      <h1>{product.name}</h1>
      <AddToCartButton productId={id} price={product.price} />
    </div>
  )
}

// Client Component（需要交互）
'use client'
function AddToCartButton({ productId, price }) {
  const [added, setAdded] = useState(false)
  return <button onClick={() => setAdded(true)}>{added ? '已加入' : '加入购物车'}</button>
}
```

#### RSC 传输协议（RSC Payload）

```
不是 HTML 字符串，是流式 JSON 格式：
J0:["$","div",null,{"children":[
  ["$","h1",null,{"children":"iPhone 15"}],
  ["$","$L1",null,{"productId":"123"}]  ← Client Component 占位符
]}]
```

#### 解决的问题

```js
// 1. Bundle 体积：marked 库不打包进客户端
async function BlogPost({ content }) {
  const html = marked(content)  // 服务端执行
  return <div dangerouslySetInnerHTML={{ __html: html }} />
}

// 2. 消除数据瀑布：并行在服务端获取
async function Page() {
  const [user, posts] = await Promise.all([db.getUser(), db.getPosts()])
  return <Profile user={user} posts={posts} />
}

// 3. 安全：DB 连接永远不到客户端
```

#### RSC + Suspense = 流式渲染

```jsx
async function Page() {
  return (
    <div>
      <Header />   {/* 快，立即流式返回 */}
      <Suspense fallback={<ProductSkeleton />}>
        <SlowProductList />   {/* 慢，独立流式返回，不阻塞 Header */}
      </Suspense>
    </div>
  )
}
```

#### 重要规则

```
✅ Server Component 可以导入 Client Component
❌ Client Component 不能导入 Server Component
✅ Server Component 可以把 Server 组件作为 children 传给 Client

// props 必须可序列化（函数、Date、Symbol 不行）
```

RSC 是 React 规范，Next.js App Router 是目前最成熟的实现。

### 9.2 RSC 执行环境要求 & 客户端组件 Hook 时机

#### 执行环境要求

```
需要：
├── Node.js 18+ / Deno / Bun / Edge Runtime
├── 支持 RSC 的框架（Next.js 13+ App Router 最成熟）
└── 构建工具支持（区分 server/client bundle）

React 单独不够！框架负责：路由、打包、运行时、传输、Hydration
```

#### 两个 Bundle

```

```

#### Client Component Hook 执行时机

```

  Server Component 执行 → 遇到 Client Component → 不执行它

```

// ❌ 服务端渲染结果 ≠ 客户端首次渲染 → Hydration mismatch

## 十、Suspense 机制

### 10.1 Suspense 如何感知子组件完成状态

#### 核心：throw Promise（被动捕获，不是轮询）

```
子组件 throw Promise → Suspense 捕获 → 显示 fallback
Promise resolve → React 重新渲染 → 显示真实内容
```

#### Suspense 概念实现

```js
function Suspense({ fallback, children }) {
  try {
    return children
  } catch (thrown) {
    if (thrown instanceof Promise) {
      thrown.then(() => forceRerender())  // resolve 后重渲
      return fallback
    }
    throw thrown  // 不是 Promise → 向上抛（ErrorBoundary 处理）
  }
}
```

#### 三种触发方式

```js
// 1. use(promise) — React 19
const user = use(userPromise)

// 2. 数据获取库（SWR/React Query）内部实现了 throw promise 协议
const { data } = useSWR('/user', fetcher)

// 3. React.lazy — 组件懒加载
const Chart = React.lazy(() => import('./Chart'))
```

#### 嵌套 Suspense 就近原则

```jsx
<Suspense fallback={<PageSkeleton />}>       {/* 外层 */}
  <Header />
  <Suspense fallback={<ListSkeleton />}>     {/* 内层 */}
    <SlowList />   {/* throw → 内层捕获，Header 不受影响 */}
  </Suspense>
  <Footer />
</Suspense>
```

#### Fiber 层面

```
throw promise → Fiber 标记 DidCapture → 向上找 Suspense Fiber
→ 渲染 fallback 子树 → 注册 promise 到 ping 队列
→ resolve → ping → 调度重渲 → 重新尝试子树渲染
```

#### throw Promise vs throw Error

```
throw Promise  → Suspense 捕获  → 显示 fallback，等待完成
throw Error    → ErrorBoundary  → 显示错误 UI，不恢复

// 同时处理：
<ErrorBoundary fallback={<Error />}>
  <Suspense fallback={<Spinner />}>
    <AsyncComponent />
  </Suspense>
</ErrorBoundary>
```

## 十一、Fiber 架构

### 11.1 Fiber 架构核心

#### 为什么需要 Fiber（React 15 的问题）

```
Stack Reconciler（React 15）：
  递归调用 → 同步、不可中断 → 大树 > 16ms → 主线程阻塞 → 掉帧

Fiber（React 16+）：
  链表结构 → 可中断/恢复 → 时间切片 → 让出主线程 → 流畅
```

#### Fiber 节点 = 工作单元

```js
{
  type, key,            // 组件信息
  stateNode,            // 对应 DOM 节点 / 组件实例
  child,                // 第一个子节点
  sibling,              // 下一个兄弟节点
  return,               // 父节点（用于回溯）
  pendingProps,         // 新 props
  memoizedProps,        // 旧 props
  memoizedState,        // hooks 链表
  flags,                // 需要的 DOM 操作（增/删/改）
  lanes,                // 优先级
  alternate,            // 指向另一棵树的对应节点（双缓冲）
}
```

#### 链表遍历：可随时中断

```
树结构：App → Header, Main → List → Item1, Item2

Fiber 链表：
App.child → Header
Header.sibling → Main
Main.child → List
List.child → Item1
Item1.sibling → Item2
所有节点.return → 父节点

遍历：处理节点 → 有子 → 去 child；无子 → 去 sibling；无兄弟 → 去 return
每处理完一个节点：检查时间，不够就停！
```

#### 时间切片（Time Slicing）

```js
const FRAME_TIME = 5  // 5ms 时间片

function workLoop() {
  while (workInProgress !== null && !shouldYield()) {
    workInProgress = performUnitOfWork(workInProgress)
  }

  if (workInProgress !== null) {
    scheduleCallback(workLoop)  // 让出，下次继续
  } else {
    commitRoot()  // 全部完成
  }
}

function shouldYield() {
  return Date.now() >= deadline
}
```

**为什么用 MessageChannel 不用 setTimeout：**
- `setTimeout(fn, 0)` 最小延迟 ≈ 4ms
- `MessageChannel.postMessage` 延迟 < 1ms，更精确

#### 双缓冲树

```
current 树（当前屏幕显示）
        ↕ alternate 指针互相连接
workInProgress 树（后台构建，可中断）

完成 → commit 阶段 → root.current = workInProgress（指针切换，瞬间）
```

#### 优先级（Lanes）

```js
SyncLane           = 0b001   // 最高，用户输入
InputContinuousLane = 0b100  // 连续输入
DefaultLane        = 0b10000 // 普通 setState
TransitionLane     = ...     // useTransition
IdleLane           = ...     // 最低，空闲时
```

#### 两阶段渲染

```
Render Phase（可中断）：
  beginWork（自顶向下）：比较新旧，执行函数组件，生成子 Fiber，标记 flags
  completeWork（自底向上）：创建 DOM 节点（不插入），冒泡 flags

Commit Phase（同步不可中断）：
  beforeMutation → mutation（操作真实 DOM）→ layout（useLayoutEffect）
```

### 11.2 高优先级打断 & 恢复低优先级渲染

#### 核心原则

```
只有 Render Phase 可被打断
Commit Phase 一旦进入，必须跑完
```

#### 完整流程

```
t0: 低优先级渲染进行中（workInProgress 第50个节点...）
t1: 用户输入 → SyncLane 更新
    → shouldYield() 返回 true → workLoop 退出
    → workInProgress 树【保留在内存，不丢弃】
t2: ensureRootIsScheduled() 发现高优先级
    → cancelCallback(低优先级任务)
    → scheduleCallback(高优先级, ImmediatePriority)
t3: 高优先级 workLoop → commitRoot → current 树更新
t4: ensureRootIsScheduled() 重新调度低优先级
t5: 低优先级从 root 重新开始（不是断点续传！）
```

#### 为什么必须重新开始，不能断点续传？

```
低优先级做到一半时 state = S1
高优先级更新后 state = S2

断点续传：前50节点基于S1，后50节点基于S2 → UI 撕裂！
必须从 root 重算，基于最新 state 保证一致性
```

#### Lane 位运算

```js
root.pendingLanes |= SyncLane  // 新更新进来

function getNextLanes(root) {
  const pending = root.pendingLanes
  return pending & -pending  // 取最低 bit = 最高优先级
}
```

### 11.3 Scheduler：MessageChannel 让出与恢复

#### MessageChannel 是宏任务

```js
const channel = new MessageChannel()

// port1 收到消息 → 作为下一个宏任务执行（不是立即！）
channel.port1.onmessage = () => {
  deadline = Date.now() + 5  // 新的 5ms 时间片
  workLoop()                  // 恢复工作
}

function workLoop() {
  while (workInProgress !== null) {
    if (shouldYield()) {
      channel.port2.postMessage(null)  // 让出！排队下一个宏任务
      return                           // 当前宏任务结束，主线程释放
    }
    workInProgress = performUnitOfWork(workInProgress)
  }
  if (workInProgress === null) commitRoot()
}
```

#### 为什么宏任务能让出主线程

```
宏任务之间，浏览器可以插入：
  ✅ 渲染（样式计算 + 布局 + 绘制）
  ✅ 用户输入处理
  ✅ requestAnimationFrame

微任务（Promise.then）不行！
  微任务在当前宏任务结束后立即清空
  浏览器没有机会插入渲染和输入处理
```

#### 时间线

```
宏任务1（workLoop，0~5ms）
  port2.postMessage(null)  ← 排队宏任务 #2
  return

【主线程空闲】
  浏览器处理：输入 / 动画 / 绘制

宏任务2（port1.onmessage）
  deadline = now + 5ms
  继续 workLoop...

宏任务N（workInProgress === null）
  commitRoot() → 完成
```

#### 高优先级如何在空闲期插队

```
宏任务1 结束，postMessage 排队 #2
空闲期：用户输入 → SyncLane
  → scheduleCallback(ImmediatePriority) → 排到 #1 位置（插前面）
  → 取消 port1.onmessage 回调
宏任务2：高优先级执行
宏任务3：低优先级重新调度，从 root 重来
```

#### 一句话总结

> `postMessage` 把"继续工作"变成宏任务重新排队，宏任务之间浏览器天然处理更重要的事（输入/绘制）。这就是"让出主线程"的本质——不是真的暂停，而是把剩余工作**重新排队**，让高优先级事情先做。
